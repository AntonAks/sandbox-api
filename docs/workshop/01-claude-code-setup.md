# 01 — Налаштування Claude Code під проект

> Навчальна замітка для workshop. Тема: як правильно засетапити Claude Code на старті проекту, до написання першого рядка коду.

## Що таке Claude Code (коротко)

Claude Code — це CLI/IDE-інструмент, в якому LLM-агент (Claude) виконує задачі в контексті твого репо: читає файли, редагує код, запускає команди, створює PR-и. Доступний як CLI, desktop-додаток, web (claude.ai/code) і розширення для VS Code/JetBrains.

Ключове, що відрізняє його від чату загального призначення: він має **інструменти**, які виконують реальні дії на твоїй машині (read/write files, bash, git тощо). Це робить продуктивність високою — і одночасно вимагає **усвідомленого налаштування**, що йому можна, а що ні.

## Чотири шари конфігурації, які треба знати

### 1. Project rules (CLAUDE.md і "пам'ять проекту")

`CLAUDE.md` у корені репо — головна точка входу для правил, конвенцій, важливих фактів про проект. Claude автоматично читає його щоразу як стартує сесію в цьому каталозі. Імпорти з інших файлів робляться через синтаксис `@path/to/file.md` всередині `CLAUDE.md` — так можна не пхати весь зміст в один великий файл.

Місця де можуть жити правила:
- `CLAUDE.md` (репо) — командні правила, в гіті, для всіх.
- `CLAUDE.md` (домашня папка `~/.claude/CLAUDE.md`) — персональні правила, не в гіті.
- Підкаталоги — `CLAUDE.md` всередині піддиректорії підвантажується тільки коли працюєш з тим піддиректорієм.

**Що писати в CLAUDE.md:** конвенції яких не видно з коду (стиль, домени, "ніколи не роби X"), посилання на ключові документи (як `docs/spec.md`), команди-shortcuts, що-не-чіпати.

**Чого НЕ писати:** того, що Claude і так бачить з коду (структура, дерево, очевидні факти з `package.json`/`pyproject.toml`).

### 2. Skills (навички)

Skills — це **спеціалізовані інструкції/процедури**, які підвантажуються тільки коли релевантні. Технічно це Markdown-файли з YAML frontmatter (`name`, `description`, `when_to_use`), які описують конкретну процедуру: "як писати тести", "як рев'ювити код", "як налагоджувати performance issue".

Як це влаштовано:
- Кожен skill — окремий файл. Список метаданих (description) Claude бачить на старті сесії.
- Коли користувач запитує щось схоже за змістом на опис скіла, Claude **викликає** його через `Skill` tool — і тоді отримує повний текст скіла з покроковими інструкціями.
- Це економить контекст: скіл займає місце в context window тільки коли потрібен.

Звідки беруться skills:
- **Plugins / superpowers** — спільнотні набори (типу `superpowers:brainstorming`, `superpowers:test-driven-development`), які ти підключаєш через plugin marketplace.
- **Custom skills** — твої власні `.md` файли в `.claude/skills/` (особисто чи в репо). Можна писати під свій workflow.

Приклади з цього сетапу: `superpowers:brainstorming` змушує спочатку спитати "а що ми взагалі будуємо" перед кодом; `superpowers:test-driven-development` примушує писати тест перед фічею; `update-config` знає як редагувати `settings.json`.

**Філософія:** skill — це "збережений рецепт того як треба робити X". Замість того щоб щоразу пояснювати Claude як рев'ювити код за твоїми стандартами — пишеш скіл один раз, і він застосовується автоматично.

### 3. Hooks (хуки)

Hooks — це **shell-команди, які Claude Code запускає у відповідь на події під час сесії**. Конфігуруються в `.claude/settings.json` (репо) або `~/.claude/settings.json` (глобально).

Важливо: хук виконує **harness** (сам Claude Code), не Claude. Це означає:
- Хук **завжди** спрацює на подію, навіть якщо LLM "забув".
- Через хуки можна зробити автоматичні правила, які LLM не зможе оминути.
- Ціна цьому — це shell-команди з твоїми правами, тому будь обережним з тим що там пишеш.

Типи подій (event triggers), найкорисніші:
- **PreToolUse** — перед викликом інструмента (можна заблокувати або модифікувати).
- **PostToolUse** — після виклику (наприклад, прогнати лінт після кожного `Edit`).
- **UserPromptSubmit** — коли користувач натиснув enter (можна підвантажити контекст).
- **SessionStart** — на старті сесії (підвантажити свіжі факти з env, git status тощо).
- **Stop** — коли Claude закінчив відповідати (нагадування, звукові сигнали).
- **Notification** — коли Claude чекає на тебе (зручно для desktop notifications).

Приклади корисних хуків:
- "Після кожного редагування .py файла — прогнати `ruff check` тільки на цьому файлі".
- "Перед тим як `git push` — переконатись що тести зелені".
- "На старті сесії — показати які гілки відстають від main".

**Не зловживати.** Кожен хук — це shell, який блокує сесію. Багато повільних хуків зроблять Claude Code тормозним.

### 4. Безпека (доступ і обмеження)

Що Claude Code **може** в цьому сетапі:
- Читати/писати файли всередині поточного робочого каталогу (`~/Documents/_MyRepos/sandbox-api`) і його піддиректорій. Доступ до інших шляхів (як `~/.ssh/`, `~/.claude/`) теж є — це частина моєї конфігурації, тому я туди ходжу свідомо коли треба.
- Виконувати shell-команди через `Bash` tool — з твоїми правами користувача.
- Робити мережеві запити (через `WebFetch`, `WebSearch`, `curl`/`wget` через bash, MCP-сервери як context7).
- Викликати під-агентів (`Task` tool) для паралельних дослідницьких задач — вони бачать тільки те, що я їм передам у промпті.
- Створювати git-коміти, push-ити, відкривати PR через `gh` — але **тільки коли ти прямо просиш**. Без явного дозволу — лише локальні зміни.

Що Claude Code **не робить без явного дозволу**:
- Не пушить нічого до remote.
- Не запускає `tofu apply`/`terraform apply` (а в цьому проекті — взагалі ніколи, бо ти робиш це руками).
- Не робить деструктивні git-операції (`reset --hard`, `push --force`, видалення гілок) без підтвердження.
- Не міняє глобальні git-настройки.
- Не пропускає pre-commit хуки через `--no-verify`.

Permission system:
- За замовчуванням `Bash` tool кожен раз питає дозволу на нову команду.
- Через `.claude/settings.local.json` можна додати allowlist для команд які ти готовий пускати без підтвердження (наприклад `npm test`, `pytest`, `git status`).
- `settings.json` — командний (в гіті), `settings.local.json` — персональний (в gitignore).

Sandbox/обмеження:
- Я **не маю** доступу до твоїх паролів менеджера, браузерної історії, чатів, інших процесів.
- Я **не маю** доступу до твого ssh-agent ключів напряму — я можу запускати `ssh`/`git push`, але приватний ключ розшифровує і використовує сама `ssh`-команда під твоїми правами.
- Все що я роблю — видно в conversation transcript і повертається назад до тебе як tool result.

## Як це підготовлено в цьому репо

Створено:
- `.claude/` — порожня директорія, готова під `settings.json` (хуки, permissions) і `skills/` (custom skills, якщо будуть).
- `docs/rules/` — порожня директорія, куди ляжуть твої md-документи з правилами проекту. Потім підключимо їх через `@docs/rules/...md` з `CLAUDE.md`.
- `docs/workshop/` — навчальні замітки (цей файл — перший).

Ще не створено (свідомо, чекаємо на твої рішення):
- `CLAUDE.md` — створимо коли побачимо твої md-документи з правилами, щоб одразу правильно їх імпортувати.
- `.claude/settings.json` — створимо разом, коли визначимось які хуки потрібні.
- `.claude/settings.local.json` — створимо разом коли побачимо які команди тобі ОК пускати без підтверджень.

## Висновки для workshop

1. **Сетап = частина пайплайна.** Хороший Claude Code workflow не в тому "як хитро попросити", а в тому **що ти налаштував один раз** в проекті: правила, хуки, скіли. Решта йде сама.
2. **Розділяй "що" і "як".** "Що" → CLAUDE.md (правила, факти, цілі). "Як" → skills (процедури) + hooks (автоматизація).
3. **Hooks > prompts** там де треба гарантія. Якщо щось має ОБОВ'ЯЗКОВО спрацювати — це хук, не нагадування в промпті.
4. **Безпека за замовчуванням сувора.** Жодних `--no-verify`, жодних `force push` без явного запиту, жодних деструктивних дій по інерції. Це особливо важливо коли demo робиться наживо перед аудиторією.
